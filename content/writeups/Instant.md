# Enumeration

The machine is hosting a web sever at port 80 and is only running `ssh`. We can find that our requests to the website are being redirected to `instant.htb`. Adding that to our `/etc/hosts` and browsing around we can find the link to download an `apk` file.

`apk` files are just zipped files containing some resources and code. They are similar to `jar` files. We can simply unzip it and look at the contents inside. The notable files here are

- `classes.dex`, This contains the main code for the application
- `resources.arsc`, contains resources and identifiers for the application like strings
- `res/*` contains binary `xml` resources like images or other strings

`classes.dex` is application code transformed into Dex byte code. We can convert it back to jar by using [dex2jar](https://github.com/pxb1988/dex2jar)

```sh
~/opt/dex-tools/d2j-dex2jar.sh -f -o classes.jar classes.dex
```

Then we can unzip the jar file and look at the code inside. There is some interesting stuff is at `com/instantlabs/instant`, We find a admin `jwt` token inside `AdminActivities.java`. We can also find the code making API calls to `mywalletv1.instant.htb` although I couldn't find any vulnerabilities or such in any endpoint referenced by the code.

We could also try cracking the `jwt` token signature to find the secret it is hashed with. This is only possible if the signing key is very weak and takes a long time [this](https://github.com/brendan-rius/c-jwt-cracker) repository can help us crack the token.

I tried this for some time but couldn't find anything and my ancient laptop was about to explode so I gave up.

I then started to look for stuff in the resource files in the `apk`. Doing a search for `instant.htb` on the files in `res/` gave me another subdomain.

```sh
strings ./* | grep instant.htb
```

with this I found `swagger-ui.instant.htb`. Adding this to `/etc/hosts` and browsing around leads us to `http://swagger-ui.instant.htb/apidocs/` which introduces a bunch of new API endpoints.

# Privilege Escalation

There is a `/opt/backups/Solar-PuTTy` folder which contains a `session.dat`. Solar Putty is `solarwinds`' version of `PuTTy` It has some extra features and a GUI also the ability to store session information for easy log in.

So if we manage to decrypt the session file found we will probably get the required credentials for root. [This](https://github.com/RainbowCache/solar_putty_crack) is very helpful. It requires `dotnet` 6.0 runtime. We can copy the `sessions.dat` file and try to crack it locally.

```sh
./sp_crack ~/Documents/Wordlists/rockyou.txt session.dat
```

And we find the password for root user

`12**24nzC!r0c%q12`
